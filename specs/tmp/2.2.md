# Task 2.2: Implement image_digest retrieval via docker inspect

## WHAT

Create a utility function `getImageDigest(exec: ExecFn, image: string): Promise<string | null>` that retrieves the content-addressable SHA256 digest of a Docker image as reported by the Docker daemon on the remote host. Returns `null` if the image is not found locally or has no digest.

The digest is retrieved via:

```
docker image inspect <image> --format '{{index .RepoDigests 0}}'
```

This returns the full digest reference like `ghcr.io/org/app@sha256:abc123...`. Fleet extracts and stores just the `sha256:abc123...` portion.

## WHY

V1.2 spec §2.3 — the `image_digest` catches changes to floating tags like `latest` where the tag string is unchanged but the image content has changed. Two images with different tags but identical content have the same digest. Comparing the stored digest (from the previous deploy) against the current digest (after pulling) determines whether a `docker pull` produced a new image and whether the service must therefore be redeployed. Without this check, a service using `image: myapp:latest` would never be redeployed when the tag is re-pushed.

## HOW

1. Add `getImageDigest(exec: ExecFn, image: string): Promise<string | null>` to `src/deploy/hashes.ts` (same file as `computeDefinitionHash`, per task 2.1).

2. Run the inspect command via `exec()`:
   ```
   docker image inspect <image> --format '{{index .RepoDigests 0}}'
   ```

3. If `result.code !== 0`, return `null` (image not present locally — inspect failed).

4. Trim the stdout. If it is empty or equals `"<no value>"`, return `null`. Docker outputs `<no value>` for images that have no `RepoDigests` entry (e.g., locally built images).

5. Parse the stdout to extract the `sha256:...` portion: split on `@` and take the second part (index 1). Trim any remaining whitespace. Return that string.

6. Edge cases to handle:
   - Local images built with `docker build` may have no `RepoDigests` — return `null`. Callers treat `null` as "no digest available" and skip digest comparison for that service, falling back to definition_hash and env_hash alone.
   - `docker pull` must be called by the deploy flow before this function so the local image reflects the latest registry content. `getImageDigest` itself does not pull.
   - If the image string contains a tag (e.g., `myapp:latest`) or a digest reference (e.g., `myapp@sha256:...`), pass it through to `docker image inspect` as-is; Docker resolves it correctly.

7. Export `getImageDigest` alongside `computeDefinitionHash` from the deploy module's barrel export.

8. Write a unit test in `tests/deploy/hashes.test.ts` (alongside the task 2.1 tests) that mocks `exec` and verifies:
   - Normal case: stdout `"ghcr.io/org/app@sha256:abc123"` → returns `"sha256:abc123"`
   - Empty stdout → returns `null`
   - Stdout `"<no value>"` → returns `null`
   - Non-zero exit code → returns `null`
   - Stdout with trailing newline is trimmed correctly
