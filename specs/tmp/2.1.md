# Task 2.1: Implement definition_hash computation utility

## WHAT

Create a new utility function `computeDefinitionHash(service: ParsedService): string` that produces a deterministic SHA256 hash of the service definition fields that affect runtime behavior. The hash is prefixed with `"sha256:"`.

Fields included in the hash (per spec §2.2):
`image`, `command`, `entrypoint`, `environment`, `ports`, `volumes`, `labels`, `user`, `working_dir`, `healthcheck`

Fields explicitly excluded (per spec §2.2):
`deploy`, `depends_on`, `restart`, `networks`, `container_name`

The algorithm:
1. Extract only the included fields from `ParsedService`
2. Remove null and undefined fields (deep clean)
3. Sort all object keys alphabetically (deep sort — ensures field ordering is irrelevant)
4. `JSON.stringify` with no extra whitespace
5. SHA256 the resulting string
6. Return `"sha256:" + hexdigest`

## WHY

V1.2 spec §2.2 — the `definition_hash` is one of the three hashes used to detect whether a service needs redeployment. If any of the included runtime fields changes between deploys, the service must be redeployed. The hash is stored in `state.json` per-service. By deterministically normalizing the field set before hashing, two logically identical service definitions always produce the same hash regardless of how the compose file was authored (key order, null vs. omitted fields, etc.).

## HOW

1. Create `src/deploy/hashes.ts` (new file, following the module pattern in `src/`). Use Node.js built-in `crypto.createHash('sha256')` — no new dependencies needed.

2. Implement a `removeNullAndEmpty(obj: unknown): unknown` helper that recursively removes `null`, `undefined`, and empty-string values from objects and arrays. For objects, omit keys whose value is null, undefined, or empty string after recursion. For arrays, filter out null/undefined/empty-string elements and recurse into remaining elements.

3. Implement a `sortKeysDeep(obj: unknown): unknown` helper that recursively sorts object keys alphabetically. Arrays keep their order — do not sort array elements, but do recurse into them to sort any nested object keys.

4. Implement `computeDefinitionHash(service: ParsedService): string`:
   - Extract only the 10 included fields: `image`, `command`, `entrypoint`, `environment`, `ports`, `volumes`, `labels`, `user`, `working_dir`, `healthcheck`
   - Build a plain object containing only those keys (omit keys that are not present on the service)
   - Call `removeNullAndEmpty`, then `sortKeysDeep`, then `JSON.stringify`
   - Pass the resulting string through `crypto.createHash('sha256').update(...).digest('hex')`
   - Return `"sha256:" + hexdigest`

5. Export `computeDefinitionHash` from `src/deploy/index.ts` (or whichever barrel file the deploy module uses for its public exports).

6. Write a unit test in `tests/` (e.g., `tests/deploy/hashes.test.ts`) that verifies:
   - (a) The same input always produces the same hash (idempotency)
   - (b) Reordering the keys in `environment` or `labels` produces the same hash (key-order independence)
   - (c) Changing `image` produces a different hash
   - (d) Changing `restart` does NOT change the hash (excluded field — must be verified explicitly)
   - (e) A null `command` and an absent `command` produce the same hash (null-cleaning)
