# 3.1 — Per-service change classification (deploy / restart / skip)

## WHAT

Create a function `classifyServices(compose, stackState, candidateHashes, envHashChanged): ServiceClassification` that evaluates each service in the compose file and returns three lists: `toDeploy: string[]`, `toRestart: string[]`, `toSkip: string[]`.

Classification logic (per spec §3 decision tree):
- One-shot services → always in `toDeploy`
- Service not in `stackState.services` → `toDeploy` (first deploy or new service)
- `definition_hash` changed → `toDeploy`
- `image_digest` changed → `toDeploy`
- `env_hash` changed (passed as param) → `toRestart`
- Nothing changed → `toSkip`

## WHY

V1.2 spec §3 — this decision tree is the heart of selective deployment. Without a clean classification step, the deploy/restart/skip logic would be tangled throughout the deploy flow. Isolating it makes the deploy flow readable and the logic independently testable.

## HOW

1. Create `src/deploy/classify.ts` (new file). Define `ServiceClassification` interface with `toDeploy`, `toRestart`, and `toSkip` string arrays.
2. Implement `classifyServices(compose: ParsedComposeFile, stackState: StackState, candidateHashes: Map<string, CandidateHashes>, envHashChanged: boolean): ServiceClassification`.
3. `CandidateHashes` is a per-service object containing: `definitionHash: string`, `imageDigest: string | null`.
4. For each service name in `compose.services`:
   a. If `isOneShot(service)` → push to `toDeploy`, continue
   b. Look up `stackState.services?.[name]`
   c. If not found → push to `toDeploy`, continue
   d. If stored `definition_hash !== candidateHashes.definitionHash` → `toDeploy`
   e. Else if stored `image_digest !== candidateHashes.imageDigest` (and both non-null) → `toDeploy`
   f. Else if `envHashChanged` → `toRestart`
   g. Else → `toSkip`
5. Export `classifyServices` and `ServiceClassification` from the module.
6. Write unit tests covering all classification cases including edge cases (null digest, missing services block).
