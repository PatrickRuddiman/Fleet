# 4.3 — Post-deploy state.json update (after deploy step 11)

## WHAT

After step 11 (selective deploy execution), update `state.json` to record the new per-service state. For each deployed or restarted service: write `image`, `image_digest`, `definition_hash`, `deployed_at` (current timestamp), `skipped_at: null`. For each skipped service: write `skipped_at` (current timestamp), leave all other fields unchanged. Update `env_hash` at the stack level if the env file changed.

## WHY

V1.2 spec §6.1 (after step 11) — `state.json` is the single source of truth for what was last deployed. If hashes are not updated after deploy, the next run will incorrectly compare against stale hashes. Recording `skipped_at` enables `fleet ps` to show operators when each service was last actively deployed vs. last seen as unchanged.

## HOW

1. Open `src/deploy/deploy.ts`. After step 11 (selective deploy), before step 14 (register routes) or wherever state is currently written (step 16):
   a. For each service in `classification.toDeploy` or `classification.toRestart`:
      - Set `ServiceState.image = service.image`
      - Set `ServiceState.image_digest = candidateHashes.get(name).imageDigest ?? '(none)'`
      - Set `ServiceState.definition_hash = candidateHashes.get(name).definitionHash`
      - Set `ServiceState.deployed_at = new Date().toISOString()`
      - Set `ServiceState.skipped_at = null`
      - Set `ServiceState.one_shot = isOneShot(service)`
   b. For each service in `classification.toSkip`:
      - Set `ServiceState.skipped_at = new Date().toISOString()`
      - Leave all other fields unchanged
   c. Set `StackState.env_hash = newEnvHash ?? existingEnvHash`
2. Merge the updated `services` map into `stackState.services` (creating it if absent).
3. Pass updated state to `writeState()` as part of the existing step 16 (write state).
4. First deploy (no prior state): all services will be in `toDeploy` (classified as not running), so all services get `deployed_at` set. The `services` map is created from scratch.
