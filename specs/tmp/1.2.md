# Task 1.2: Extend state.json schema for per-service tracking

## WHAT

Add two new interfaces to `src/state/types.ts`:

1. `ServiceState` with fields:
   - `one_shot: boolean`
   - `image: string`
   - `image_digest: string`
   - `definition_hash: string`
   - `deployed_at: string`
   - `skipped_at: string | null`

2. Extend `StackState` with:
   - `env_hash?: string` — stack-level hash of .env file content
   - `services?: Record<string, ServiceState>` — per-service tracking map

Both new fields on `StackState` are optional (`?`) so existing stacks without the data remain valid. First-deploy behavior treats missing entries as "not yet deployed".

## WHY

V1.2 spec §5 defines the full per-stack and per-service state structure. This is the foundation for all change detection — hashes are stored here and compared on each deploy. Without this schema change, no other V1.2 feature can persist its results. The optional fields ensure backward compatibility with existing state.json files written before V1.2, preventing crashes on upgrade.

## HOW

1. Open `src/state/types.ts`. Add the `ServiceState` interface above or near the `StackState` interface. All 6 fields are required on `ServiceState` (not optional) — a `ServiceState` record is only written when a service has been deployed at least once.

2. Add `env_hash?: string` and `services?: Record<string, ServiceState>` to the existing `StackState` interface.

3. Open `src/state/state.ts`. Review the `readState` function (or equivalent deserialization path) to confirm it does not throw if `services` or `env_hash` are absent from a parsed state.json. These fields should be `undefined` when missing, not an error. TypeScript's optional field semantics handle this automatically as long as no code accesses them without a null/undefined guard — do a quick search for any direct `stackState.services.` accesses that lack an optional chain (`?.`).

4. No changes to `writeState` are needed — it serializes whatever `FleetState` object is passed in via `JSON.stringify`, so newly added fields will appear in the output automatically once populated.

5. Update any TypeScript tests or fixtures in `tests/` that construct `StackState` literals. Because the new fields are optional, no existing test is forced to change. Add a new test asserting that `readState` does not throw when given a state.json that lacks `services` and `env_hash`, and that accessing `state.stacks['x'].services` returns `undefined` rather than crashing.
