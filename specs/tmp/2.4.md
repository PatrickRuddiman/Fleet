# Task 2.4: Implement one-shot service detection

## WHAT

Add a query function `isOneShot(service: ParsedService): boolean` to `src/compose/queries.ts`. Returns `true` if the service's `restart` policy is `"no"` or `"on-failure"` (with or without a max-attempts suffix). Returns `false` for `"unless-stopped"`, `"always"`, or when `restart` is `undefined` (which defaults to `"unless-stopped"` behavior in Swarm mode).

Also add a convenience function `getOneShots(compose: ParsedComposeFile): string[]` that returns the names of all one-shot services in a compose file.

## WHY

V1.2 spec §2.5 — one-shot services (migration jobs, seed runners, test fixtures, etc.) must always run on every deploy regardless of change detection. Long-running services are only redeployed when one of their three hashes has changed. The deploy flow must classify each service before deciding what to do with it. Centralizing this classification in `queries.ts` keeps it testable and reusable across the deploy flow and any future commands that need to distinguish service types.

## HOW

1. Open `src/compose/queries.ts`. Add `isOneShot(service: ParsedService): boolean` with the following logic:
   - If `service.restart` is `undefined` or `null` → return `false` (long-running default)
   - If `service.restart === "no"` → return `true`
   - If `service.restart` starts with `"on-failure"` → return `true` (covers both `"on-failure"` and `"on-failure:3"` short-form syntax)
   - Otherwise → return `false`

2. This function requires `restart` to be present on `ParsedService`. Task 1.1 is a prerequisite — it adds `restart` (and other fields) to the `ParsedService` type in `src/compose/types.ts`.

3. Add `getOneShots(compose: ParsedComposeFile): string[]` that iterates over the services in the compose file and returns the names (keys) of all services for which `isOneShot` returns `true`. The return value is a list of service name strings, not service objects.

4. Export both functions from `src/compose/queries.ts`. If `queries.ts` does not yet exist, create it as a new module in `src/compose/` following the existing module conventions in that directory.

5. Write unit tests in `tests/compose/queries.test.ts` (or alongside existing compose tests) covering all classification cases for `isOneShot`:
   - `restart` is `undefined` → `false`
   - `restart` is `"no"` → `true`
   - `restart` is `"on-failure"` → `true`
   - `restart` is `"on-failure:3"` → `true`
   - `restart` is `"unless-stopped"` → `false`
   - `restart` is `"always"` → `false`

   And for `getOneShots`:
   - A compose file with a mix of one-shot and long-running services returns only the one-shot names
   - A compose file with no one-shot services returns an empty array
