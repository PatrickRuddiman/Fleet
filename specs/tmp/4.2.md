# 4.2 — Selective deploy execution — compose mode (modify deploy step 11)

## WHAT

Modify step 11 (`docker compose up`) in `src/deploy/deploy.ts` to deploy selectively. Instead of running `docker compose up -d` for all services, split into three operations based on `ServiceClassification`:
- `toDeploy` services: run `docker compose up -d <service>` per service (or batched)
- `toRestart` services: run `docker compose restart <service>` per service (no image pull)
- `toSkip` services: log skip message, do nothing

In `--force` mode: run `docker compose up -d --remove-orphans` for all services (existing behavior).

## WHY

V1.2 spec §6.1 (modified step 11) — this is the core selective deployment behavior. Databases and caches that haven't changed are not restarted. Only changed services are redeployed. `docker compose restart` for env-only changes avoids an image pull while still restarting the container to pick up new env vars.

## HOW

1. Open `src/deploy/deploy.ts`. Locate step 11 (`docker compose up -d`).
2. Check `options.force`:
   - If true: run `docker compose up -d --remove-orphans` (all services) — same as today.
   - If false: iterate `classification.toDeploy` and run `docker compose up -d <name>` for each. Then iterate `classification.toRestart` and run `docker compose restart <name>` for each.
3. For each skipped service in `classification.toSkip`, log: `  ⊘ <service> — no changes detected, skipped`.
4. After deploy commands complete, run `getImageDigest` for each deployed service to capture the actual digest of the running image. Store in `candidateHashes`.
5. V1.5 Swarm mode note (for future): in Swarm mode, `docker compose up -d <service>` becomes `docker service update --image <image> <stack>_<service>`, and `docker compose restart <service>` becomes `docker service update --force <stack>_<service>`. Document this in the HOW but do not implement Swarm mode changes in this task.
