# Task 7.1: fleet validate: warn on restart:on-failure without max_attempts

## WHAT

Add a new validation check to `src/validation/compose-checks.ts` that warns when a service has `restart: on-failure` but does not set a `max_attempts` limit. Add a new code `ONE_SHOT_NO_MAX_ATTEMPTS` to the `Codes` enum in `src/validation/types.ts`. The check is a warning (not an error) — validation passes and deploy proceeds regardless.

## WHY

V1.2 spec §8 notes that `restart: on-failure` without `max_attempts` can cause a failed one-shot job to restart indefinitely. Fleet should surface this to the operator before it becomes a production incident. A warning at `fleet validate` time gives operators the opportunity to set a bound before deploying, without blocking the deploy when they have intentionally omitted it.

## HOW

1. Open `src/validation/types.ts`. Add `ONE_SHOT_NO_MAX_ATTEMPTS = 'ONE_SHOT_NO_MAX_ATTEMPTS'` to the `Codes` enum.

2. Open `src/validation/compose-checks.ts`. Add a new exported function `checkOneShotNoMaxAttempts(compose: ParsedComposeFile): Finding[]`. The function iterates over all services in `compose.services`. For each service where `service.restart === 'on-failure'`, check whether `service.deploy?.restart_policy?.max_attempts` is absent or undefined. If so, push a `Finding` with:
   - `severity: 'warning'`
   - `code: Codes.ONE_SHOT_NO_MAX_ATTEMPTS`
   - A message matching the spec §8 example — e.g., `"Service 'NAME' uses restart: on-failure without a max_attempts limit — a failing job will restart indefinitely"`
   - A resolution string suggesting `deploy.restart_policy.max_attempts: 3` as a starting point

3. Open `src/validation/index.ts` (or whichever file defines `runAllChecks`). Add `checkOneShotNoMaxAttempts` to the list of check functions that `runAllChecks` invokes, and merge its returned `Finding[]` into the combined results.

4. This check requires `restart` to be present on `ParsedService` — task 1.1 (compose parser extension) must be implemented first. Additionally, the check reads `service.deploy?.restart_policy?.max_attempts`, which means `ParsedService` must also expose the `deploy` field (or at minimum `deploy.restart_policy.max_attempts`). Either extend task 1.1 to capture `deploy` from the raw YAML service object, or capture only `deploy?.restart_policy?.max_attempts` as a standalone optional number field `restartPolicyMaxAttempts?: number` on `ParsedService`. Coordinate with the task 1.1 implementer to agree on the approach before writing this check.

5. Add a test in `tests/` following the pattern of existing compose-check tests. Cover at least three cases: (a) service with `restart: on-failure` and no `deploy` block — expect one warning; (b) service with `restart: on-failure` and `deploy.restart_policy.max_attempts: 3` — expect no findings; (c) service with a different restart policy such as `restart: always` — expect no findings from this check.
