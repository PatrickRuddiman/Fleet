# Task 7.2: fleet ps: show per-service last-deployed vs last-skipped timestamps

## WHAT

Update `fleet ps` output to show when each service was last actually deployed versus last skipped. Add a `LAST DEPLOYED` column (or extend the existing `DEPLOYED AT` column) to the table in `src/ps/ps.ts`. When a service has been skipped on the most recent deploy, annotate the entry to distinguish it from services that were actively deployed — for example "skipped N days ago".

## WHY

V1.2 spec §10 Open Question #4 (decided: yes) — operators need to know that `postgres` hasn't been touched in 4 days while `api` was deployed an hour ago. Without this visibility, selective deployment is opaque and operators cannot tell if a service was intentionally skipped or accidentally missed. This column makes the state of each service legible at a glance.

## HOW

1. Open `src/ps/types.ts`. Add `deployedAt: string` and `skippedAt: string | null` to `StackRow` (or the equivalent per-service row type). `deployedAt` comes from `ServiceState.deployed_at` in state.json; `skippedAt` comes from `ServiceState.skipped_at`.

2. Open `src/ps/ps.ts`. In the function that builds row objects, read `state.stacks[stackName].services?.[serviceName]` to retrieve `deployed_at` and `skipped_at` for each service. Apply the following display logic:
   - If `skipped_at` is non-null and more recent than `deployed_at`, format the cell as something like `"deployed X ago (skipped Y ago)"` to make the skip explicit.
   - If `skipped_at` is null or older than `deployed_at`, display only `deployed_at` formatted as a relative time (e.g., "2 hours ago").
   - Use a shared relative-time helper if one already exists in the codebase; otherwise add a small local utility.

3. Update the table formatter to include the new column. Follow the existing `formatTable` alignment pattern — measure the widest value in the column to set its width, and left-align text content.

4. If `services` is absent from state (old state.json written before V1.2), fall back gracefully: show only the stack-level `deployed_at` as before, or show a dash if that is also unavailable. No breaking change — `fleet ps` must remain functional on pre-V1.2 state files.

5. Update tests in `tests/` for ps output to cover the new column. Add cases for: (a) a service with only `deployed_at` set (no skip); (b) a service where `skipped_at` is more recent than `deployed_at`; (c) a stack with no `services` map (old state format) to confirm the fallback renders without error.
