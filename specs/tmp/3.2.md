# 3.2 — Selective image pull logic (modify deploy step 10)

## WHAT

Modify the image pull step in `src/deploy/helpers.ts` (currently step 10 — pulls all images unconditionally unless `--skip-pull`) to only pull images for services that need it. Add a new helper `pullSelectiveImages(exec, compose, toDeploy, force)` that:
- In `--force` mode: pulls all images
- Otherwise: pulls images for `toDeploy` services and one-shots, skips `toRestart` and `toSkip` services
- For services with floating tags (`latest` or digest-based): always pulls to get the current digest, even if the service is tentatively classified as unchanged

## WHY

V1.2 spec §6.1 (modified step 10) — pulling images for services that haven't changed wastes time and bandwidth. Selective pulling is especially important for stacks with many services where only one or two change per deploy.

## HOW

1. Open `src/deploy/helpers.ts`. Add a new exported function `pullSelectiveImages(exec: ExecFn, compose: ParsedComposeFile, toDeploy: string[], force: boolean): Promise<void>`.
2. If `force` is true: run `docker compose pull` (pull all) and return.
3. Otherwise: for each service in `toDeploy`, run `docker compose pull <serviceName>`. Services in `toRestart` and `toSkip` are skipped.
4. Special case for floating tags: if a service's `image` tag is `latest` or the image reference includes `@sha256:`, pull it regardless of classification. After pulling, `getImageDigest` can be called to check if the digest changed — if unchanged, the service stays in `toSkip`/`toRestart` (the pull was "free" verification).
5. Log a line per service: `⊘ <service> — skipped pull (no changes detected)` for skipped services.
6. The existing `--skip-pull` flag still skips ALL pulls when set.
