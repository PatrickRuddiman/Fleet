# 4.1 — Pre-deploy hash computation step (insert before deploy step 10)

## WHAT

Insert a new step into the deploy flow in `src/deploy/deploy.ts` between step 9 (secrets resolved) and step 10 (pull images): compute the current `definition_hash` and `env_hash` for all services, compare against stored state, and produce a `candidateHashes` map and `envHashChanged` boolean that later steps consume.

## WHY

V1.2 spec §6.1 — hashes must be computed before image pulls happen, so the pull step knows which services to pull. This step is purely local computation (except `env_hash` which needs to run `sha256sum` on the remote server after the `.env` file is uploaded). Isolating it as a named step keeps the deploy flow readable.

## HOW

1. Open `src/deploy/deploy.ts`. After step 9 (resolveSecrets), add a new step:
   a. For each service in `context.compose.services`, call `computeDefinitionHash(service)` → build `candidateHashes: Map<string, { definitionHash: string }>`.
   b. Compute `newEnvHash` by calling `computeEnvHash(exec, stackDir + '/.env')` (runs on remote server after `.env` is uploaded).
   c. Compare `newEnvHash` against `stackState.env_hash` → set `envHashChanged: boolean`.
   d. Store `candidateHashes` and `envHashChanged` in `DeployContext` (or as local variables passed to subsequent steps).
2. The `image_digest` is not computed here — it requires pulling the image first (step 10). Digests are computed after selective pull.
3. Pass `candidateHashes` and `envHashChanged` to `classifyServices` (task 3.1) after step 10.
4. Add `candidateHashes` and `envHashChanged` to `DeployContext` in `src/deploy/types.ts` if needed, or thread them as parameters.
