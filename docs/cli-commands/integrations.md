# Integrations: Operational Commands

Fleet's operational commands interact with several external systems and internal
modules. This page documents each integration, how it is used, and how to
troubleshoot it.

## Docker Compose

### How Fleet uses Docker Compose

Every operational command executes Docker Compose commands over SSH on the remote
server. Fleet uses the **V2 plugin syntax** (`docker compose`, not
`docker-compose`), which requires Docker Compose to be installed as a Docker CLI
plugin on the remote server.

| Command | Docker Compose invocation |
|---------|--------------------------|
| `fleet logs` | `docker compose -p <stack> logs -f [--tail N] [service]` |
| `fleet ps` | `docker compose -p <stack> ps --format json` |
| `fleet restart` | `docker compose -p <stack> restart <service>` |
| `fleet stop` | `docker compose -p <stack> stop` |
| `fleet teardown` | `docker compose -p <stack> down [--volumes]` |

All commands pass the stack name as the `-p` (project name) flag. This must
match the project name used during `fleet deploy`.

### Docker Compose version requirements

Fleet requires Docker Compose V2 (the `docker compose` CLI plugin). Verify
installation on the remote server:

```bash
ssh <server> "docker compose version"
# Expected: Docker Compose version v2.x.x
```

If you see "docker: 'compose' is not a docker command", you need to install the
Compose plugin. See the
[Docker Compose installation guide](https://docs.docker.com/compose/install/).

### Project naming

Stack names are validated against the pattern `^[a-z\d][a-z\d-]*$` (defined in
`src/config/schema.ts:46`). This ensures compatibility with Docker Compose
project naming rules, which require lowercase alphanumeric characters and
hyphens. See the
[Configuration Schema Reference](../configuration/schema-reference.md#stack)
for full stack name constraints.

### JSON output parsing

The `ps` command relies on `docker compose ps --format json`, which outputs one
JSON object per line. The parser (`src/ps/ps.ts:16-41`) handles edge cases:

- Silently skips non-JSON lines (Docker warnings, ANSI escape sequences)
- Falls back to `parsed.Name` if `parsed.Service` is missing
- Returns `"unknown"` for completely unrecognizable entries

If `docker compose ps` produces unexpected output on the remote server,
troubleshoot by running the command directly:

```bash
ssh <server> "docker compose -p <stack> ps --format json"
```

Look for non-JSON lines mixed into the output, which could indicate Docker
version incompatibilities or terminal configuration issues.

## Caddy Reverse Proxy (Admin API)

### How Fleet uses Caddy

Fleet manages Caddy routes through the
[Caddy Admin API](https://caddyserver.com/docs/api), which listens at
`http://localhost:2019` inside the `fleet-proxy` container. Because this port is
not exposed to the host, Fleet accesses it via `docker exec` on the remote
server.

The `stop` and `teardown` commands remove Caddy routes before stopping
containers. See [Teardown Operation](../stack-lifecycle/teardown.md) for the
full teardown flow and [Stop Operation](../stack-lifecycle/stop.md) for stop
details. The `restart` command does not modify Caddy routes because the
container retains its network identity after restart.

### Route identification

Routes are identified by a Caddy `@id` field using the pattern
`{stackName}__{serviceName}` (double underscore separator). This ID is
generated by `buildCaddyId()` in `src/caddy/commands.ts:11-13`.

Route removal uses the Caddy `/id/{caddyId}` endpoint:

```
DELETE http://localhost:2019/id/{stackName}__{serviceName}
```

### How to query Caddy directly

To inspect the current proxy configuration from outside Fleet:

```bash
# View all routes
ssh <server> "docker exec fleet-proxy curl -s http://localhost:2019/config/apps/http/servers/fleet/routes" | jq .

# View full configuration
ssh <server> "docker exec fleet-proxy curl -s http://localhost:2019/config/" | jq .

# Check if a specific route exists
ssh <server> "docker exec fleet-proxy curl -s http://localhost:2019/id/<stack>__<service>"

# Manually remove a route
ssh <server> "docker exec fleet-proxy curl -s -f -X DELETE http://localhost:2019/id/<stack>__<service>"
```

### What happens when the Caddy container is down

If the `fleet-proxy` container is stopped or crashed when `stop` or `teardown`
tries to remove routes, the `docker exec` call fails with a non-zero exit code.
The command throws an error and aborts. Containers remain running.

To recover:

1. Check Caddy container status:
   `ssh <server> "docker inspect fleet-proxy --format '{{.State.Status}}'"`
2. If Caddy needs to be restarted, use `fleet deploy` to trigger a bootstrap
3. Alternatively, skip route removal and manually stop containers:
   `ssh <server> "docker compose -p <stack> stop"`

### Route persistence in Caddy

Caddy stores configuration in memory by default. Routes survive container
restarts only because Fleet starts Caddy with `caddy run --resume`, which
reloads the last saved configuration from the Caddy data volume.

If the Caddy data volume is lost, all route configuration is lost. Use
`fleet deploy` to redeploy stacks, which will re-register routes.

## SSH Connection Layer

### How Fleet uses SSH

All remote operations are executed through the SSH connection layer defined in
`src/ssh/`. The connection provides two execution modes:

- **`exec(command)`**: Runs a command and returns `{ stdout, stderr, code }`
- **`streamExec(command, callbacks)`**: Streams output in real-time (used by
  `fleet logs`)

### Authentication

The SSH connection is configured through `fleet.yml`'s `server` section:

```yaml
server:
  host: "example.com"    # Required
  port: 22               # Default: 22
  user: "root"           # Default: "root"
  identity_file: "~/.ssh/id_rsa"  # Optional
```

When `identity_file` is not specified, the SSH library uses the system's default
SSH agent or default key locations. See [SSH Connection Layer](../ssh-connection/overview.md)
for the full authentication flow and [SSH Authentication](../ssh-connection/authentication.md)
for SSH agent setup.

### Troubleshooting SSH failures

Common issues:

| Symptom | Cause | Fix |
|---------|-------|-----|
| "Connection refused" | SSH not running or wrong port | Verify `host` and `port` in `fleet.yml` |
| "Permission denied" | Wrong key or user | Check `user` and `identity_file` |
| "Host key verification failed" | Unknown host | Add server to `known_hosts` |
| Command timeout | Network issues | Check network connectivity |

To verify connectivity manually:

```bash
ssh -i <identity_file> -p <port> <user>@<host> "echo ok"
```

### How `fleet logs` handles streaming

The `logs` command uses `streamExec` instead of `exec` to pipe Docker log
output directly to the terminal. It registers a `SIGINT` handler that
closes the SSH connection when Ctrl+C is pressed. The handler is removed in a
`finally` block to prevent listener leaks.

**Source**: `src/logs/logs.ts:39-58`

## Fleet State File

### How operational commands use state

Every command reads `~/.fleet/state.json` from the remote server. Commands use
state for different purposes:

| Command | Read state | Write state | State usage |
|---------|-----------|-------------|-------------|
| `logs` | Yes | No | Validate stack exists |
| `ps` | Yes | No | Route mappings, timestamps |
| `restart` | Yes | No | Validate stack exists |
| `stop` | Yes | Yes | Route IDs for Caddy removal, then remove stack |
| `teardown` | Yes | Yes | Route IDs for Caddy removal, then remove stack |

### State file location and format

- **Path**: `~/.fleet/state.json` on the remote server
- **Format**: JSON, validated by a Zod schema
- **Write method**: Atomic (write to `.tmp` file, then `mv`)

See [Server State Management](../state-management/overview.md) for the complete schema
and read/write mechanics, and
[State Lifecycle](../state-management/state-lifecycle.md) for how state flows
across all operations.

### Manual state inspection

```bash
# View current state
ssh <server> "cat ~/.fleet/state.json" | jq .

# View specific stack
ssh <server> "cat ~/.fleet/state.json" | jq '.stacks["myapp"]'

# List all stack names
ssh <server> "cat ~/.fleet/state.json" | jq '.stacks | keys'
```

### State recovery

- **Missing file**: Returns default empty state (safe, no stacks known)
- **Corrupted JSON**: Throws error -- delete the file and redeploy
- **Stale entries**: Edit manually with `jq` or a text editor

## Commander.js (CLI Framework)

### How Fleet uses Commander.js

Each command file in `src/commands/` exports a `register(program: Command)`
function that adds a subcommand to the Commander.js program instance. These
registration functions are called by the main CLI entry point (`src/cli.ts`).

The command files handle:

- Argument and option declaration
- Input validation (e.g., `--tail` must be a positive integer)
- Error catching with formatted output and `process.exit(1)`

### Adding global options

Commander.js supports global options on the parent program. To add options like
`--verbose` or `--dry-run` across all operational commands, define them on the
`program` instance in `src/cli.ts` before registering subcommands. See the
[Commander.js documentation](https://github.com/tj/commander.js#options) for
details on option inheritance.

### Subcommand name conflicts

Commander.js throws an error if two subcommands are registered with the same
name. All Fleet commands have unique names (`logs`, `ps`, `restart`, `stop`,
`teardown`, etc.), so conflicts are not currently possible unless a new command
reuses an existing name.

For a deeper look at Commander.js integration including the program factory
pattern and global error handling, see
[CLI Entry Point Integrations](../cli-entry-point/integrations.md#commanderjs).
